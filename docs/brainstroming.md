## 执行摘要

### 主题
探讨 SlowPoke 的核心设计哲学与顶层理念

### 核心发现
经过深入讨论，SlowPoke 的本质定位发生了重大澄清：

**SlowPoke 不是：**
- ❌ 传统的项目管理工具
- ❌ 团队协作平台

**SlowPoke 是：**
- ✅ 个人 TODO 管理工具（核心身份）
- ✅ 可扩展平台（类似 MCP）
- ✅ 辅助完成多维度交叉项目的工作记录系统

### 使用的技巧
- 第一性原理思考
- 概念澄清
- 边界探索
- 本质追问

## 核心哲学体系

### 一、定位哲学

#### 1.1 核心身份
**SlowPoke = 个人 TODO 管理工具**

- 核心服务对象：个人
- 核心场景：管理自我任务进度
- 核心目标：辅助完成各种个人项目和团队项目

#### 1.2 技术定位
**SlowPoke = 可扩展平台**

- 不是某个具体的 UI 工具
- 而是一个平台无关、场景无关的底层系统
- 核心定义最小化，功能通过扩展实现
- 可轻松嵌套进任意工具之中

#### 1.3 项目管理的位置
**项目管理 = 副产品**

- TODO List ≈ 项目管理
- 项目管理是多个 TODO 自然汇聚的结果
- 不是起点，而是涌现

#### 1.4 多维度支持
**应对现实工作的多维度交叉**

用户的工作场景横跨：
- 硬件项目 + 软件项目
- 私有项目 + 公司项目
- 个人项目 + 外包项目
- 等等...

传统工具强制单一组织维度，SlowPoke 支持任意维度的自由组合。

---

### 二、本体哲学

#### 2.1 TODO 的双重本质

**TODO = 原子单位**
- 逻辑上独立的实体
- 不从属于任何容器
- 是关系的端点
- 是系统的最小单元

**TODO = 工作上下文容器**
- 承载任务本身的描述
- 承载工作过程中的**资料**（收集的文档、链接、文件）
- 承载**问题**（遇到的 issue、障碍、疑问）
- 承载**思考**（反思、决策理由、权衡）
- 承载**计划**（下一步、时间线）
- 承载**关系**（影响了什么、依赖什么）

**两者不矛盾：**
- 原子性指逻辑独立性，不是内容简单性
- 就像文件是文件系统的原子，但文件内可以有丰富内容

#### 2.2 项目的本质

**概念层（用户视角）：**
- 项目 ≠ TODO
- 项目是"大粒度的目标"
- 需要更大的完成时间、更多的人员、更多的产出
- 特殊性足以让它单独作为一列，拥有固定字段

**实现层（系统视角）：**
- 项目从实现本质上依旧是 TODO
- 可以理解为继承自 TODO
- 复用 TODO 的基础能力

**关系层（运算逻辑）：**
- 项目对 TODO 可以执行"包含"运算
- 可以查询：某项目包含哪些 TODO

#### 2.3 结构哲学

**不是树形层级，而是图结构**

传统工具的失败：
- 强制树形层级（项目 → 任务 → 子任务）
- 强制单一归属（一个任务只能属于一个项目）

SlowPoke 的设计：
- TODO 是独立原子
- 结构来自关系
- 关系通过运算定义

**运算类型包括但不限于：**
- 包含（项目包含 TODO）
- 依赖（TODO A 依赖 TODO B）
- 阻塞（TODO C 阻塞 TODO D）
- 关联（TODO E 关联 TODO F）
- 子任务（TODO G 是 TODO H 的子任务）
- 过滤查询（状态=完成 的 TODO）
- 聚合（某标签下的所有 TODO）
- 交集（多个条件的组合）

**灵活挂载：**
- TODO 可以独立存在
- TODO 可以挂在项目下
- TODO 可以挂在另一个 TODO 下
- TODO 可以挂在项目的某个 TODO 下
- 同一个 TODO 可以同时属于多个维度（项目、领域、属性等）

### 三、架构哲学

#### 3.1 数据主权
**我的数据我做主**

- 本地优先存储
- 数据源头在个人设备
- 服务器是可选的中转站，不是唯一真相源

#### 3.2 选择性同步
**类似 Git + GitHub 模型**

- 本地修改 TODO → 可以选择是否推送
- 推送 → 我的其他设备可以收到
- 不推送 → 保持本地私密
- 每个 TODO 可以独立决定：是否推送、推送给谁、什么时候推送

**示例场景：**
- 手机修改了 TODO，可以选择推送或不推送
- 电脑和 web 完全同步
- 三个人的不同 TODO 都同步到服务器，服务器作为总进度展示工具
- 每个人选择哪些 TODO 展示给谁

#### 3.3 服务器的角色
**中心调度 + 分布式架构**

服务器的作用：
- 作为扩展方式
- 作为中心调度的控制器
- 收集、聚合、展示被推送的数据
- 但不是唯一真相源，本地才是

#### 3.4 操作留痕
**无条件信任 + 记录一切**

- 无条件信任团队成员
- 但每个人的操作都应该留下记录
- 透明通过记录实现
- 不需要复杂的审批、权限、评估机制

---

### 四、价值哲学

#### 4.1 记录 > 管理
**核心价值在于记录，而非控制**

- 传统工具关注"控制"（谁做什么、什么时候做、进度如何）
- SlowPoke 关注"记录"（发生了什么、谁做了什么）
- 记录是工作的自然一部分，不是额外负担

#### 4.2 AI 时代的记录哲学
**充分记录，留给 AI 索引**

传统时代：
- 记录太多 → 难以检索 → 信息过载
- 所以要精简、结构化、分类

AI 时代：
- 记录再多 → AI 可以索引 → 信息变成资产
- 所以应该尽可能记录

**AI 的角色：**
- 当前：只是未来的可能性
- 不依赖 AI，记录本身就有价值
- 但为未来 AI 增强做好准备

#### 4.3 时间哲学
**TODO 是历史的一部分**

传统工具的时间观：
- 完成 = 终点 = 消失
- TODO 是"待办清单"，做完就没了
- 聚焦"未来"（还没做的事）

SlowPoke 的时间观：
- 完成 = 状态转变 = 历史的一部分
- TODO 是"工作记录"，永久存在
- **"TODO 只是走完了它的路，但路依旧在"**
- 聚焦"全程"（过去、现在、未来）

## 核心洞察

### 洞察 1：真正的核心需求
**"只有我知道我要什么"**

这意味着：
- 工具不应预设工作方式
- 工具不应强制结构
- 工具不应替我做决定
- 工具应该适应我，而不是我适应工具

因此 SlowPoke 需要：
- 最小核心
- 最大灵活性
- 足够的可定义性和可扩展性

### 洞察 2：为什么需要 TODO 管理
**辅助完成工作，而不只是记录**

核心目的：
- 管理自我任务进度
- 辅助完成各种个人项目和团队项目
- 看清全貌、做出选择、不遗忘、记录历史

### 洞察 3：技术要求的哲学根源
**多端 + 本地存储 = 数据主权**

- 多端：在电脑、手机、平板等任何地方都能访问
- 本地存储：数据属于我，不依赖云服务
- 这些不只是技术要求，而是"数据主权"哲学的体现

### 洞察 4：现实工作的复杂性
**多维度交叉，不是单一层级**

现实场景：
- 一个 TODO 可能既属于"SlowPoke 项目"，又属于"学习 C#"
- 一个 TODO 可能横跨硬件和软件
- 一个 TODO 可能是公司项目，也是个人成长

传统工具强制选择单一维度，SlowPoke 支持任意关联。

### 洞察 5：扩展性的本质
**核心最小化，能力通过扩展实现**

类比 MCP：
- 核心只定义 TODO 的基本概念和操作
- 扩展提供丰富功能（自定义字段、自定义操作、集成外部服务）
- 用户可以根据需要自由扩展

## 核心问题的澄清

### 问题：激励/公平问题重要吗？
**答案：不重要**

原因：
- SlowPoke 的本质是个人 TODO 管理工具
- 项目管理只是副产品
- 激励、公平这些问题属于团队管理，不是核心关注点

### 问题：时间管理和冲突解决
**答案：属于实现层面**

- 同步前验证一致性即可
- 不属于顶层哲学讨论

### 问题：权限设计
**答案：由扩展实现**

- 扩展性要求很高
- 权限应该是被扩展出来的，而不是被核心设计出来的

### 问题：TODO 能删除吗？
**答案：不重要**

- 可以删除（数据主权）
- 但哲学上不鼓励
- 是否删除由用户决定

## 总结

SlowPoke 的核心哲学可以用一个公式表达：

**SlowPoke = (TODO 原子 + 工作上下文) × 关系运算 × 可扩展平台 × 本地优先 × 记录为王**

这不是一个传统的项目管理工具，而是一个：
- 以个人为中心
- 以 TODO 为原子
- 以记录为价值
- 以扩展为能力
- 以数据主权为基础

的下一代个人工作管理系统。
